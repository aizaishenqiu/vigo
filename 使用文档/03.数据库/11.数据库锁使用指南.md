## 概述

Vigo 框架提供三种数据库锁机制，应对不同的并发场景：

| 锁类型 | 适用场景 | 性能 | 冲突处理 |
|--------|----------|------|----------|
| **原子操作** | 计数器、库存扣减 | ⭐⭐⭐⭐⭐ | 数据库层面保证 |
| **乐观锁** | 配置修改、订单状态 | ⭐⭐⭐⭐ | 版本号检查 |
| **分布式锁** | 多实例同步、定时任务 | ⭐⭐⭐ | Redis 互斥锁 |

## 原子操作

### 功能说明

原子操作通过数据库层面的 `UPDATE ... SET column = column + ?` 实现，无需显式加锁，性能最优。

### API 接口

```go
// Query 层原子操作
Increment(column string, amount int64) (int64, error)      // 整数递增
Decrement(column string, amount int64) (int64, error)      // 整数递减
IncrementBy(column string, amount float64) (int64, error)  // 小数递增
DecrementBy(column string, amount float64) (int64, error)  // 小数递减

// Model 层原子操作
Increment(column string, amount interface{}) error         // 通用递增
Decrement(column string, amount interface{}) error         // 通用递减
```

### 使用示例

#### 1. 库存扣减（防止超卖）

```go
// 方式 1：使用 Query
affected, err := db.NewQuery(db.GlobalDB).
    Table("products").
    Where("id", productID).
    Decrement("stock", 1)

if err != nil {
    return err
}

if affected == 0 {
    return fmt.Errorf("库存不足")
}

// 方式 2：使用 Model
product := model.New("products").Find(productID)
err := product.Decrement("stock", 1)
if err != nil {
    return err
}
```

#### 2. 计数器递增

```go
// 文章阅读量 +1
model.New("articles").
    Where("id", articleID).
    Increment("view_count", 1)

// 用户积分 +10
model.New("users").
    Where("id", userID).
    Increment("points", 10)
```

#### 3. 余额扣减

```go
// 小数扣减
affected, err := db.NewQuery(db.GlobalDB).
    Table("users").
    Where("id", userID).
    Where("balance", balance).  // 余额检查
    DecrementBy("balance", 99.99)

if err != nil {
    return err
}

if affected == 0 {
    return fmt.Errorf("余额不足")
}
```

#### 4. 事务中原子操作

```go
err := db.Transaction(func(q *db.Query) error {
    // 1. 扣减库存
    affected, err := q.Table("products").
        Where("id", productID).
        Decrement("stock", 1)
    
    if err != nil {
        return err
    }
    
    if affected == 0 {
        return fmt.Errorf("库存不足")
    }
    
    // 2. 创建订单
    _, err = q.Table("orders").
        Insert(map[string]interface{}{
            "user_id":     userID,
            "product_id":  productID,
            "created_at":  time.Now(),
        })
    
    return err
})

if err != nil {
    // 事务回滚，库存自动恢复
    log.Printf("订单创建失败：%v", err)
}
```

### 性能指标

| 操作 | 延迟 | QPS | 说明 |
|------|------|-----|------|
| Increment | 0.5-1ms | 5,000+ | 单条 SQL |
| Decrement | 0.5-1ms | 5,000+ | 单条 SQL |
| 事务中 | 1-2ms | 2,000+ | 包含事务开销 |

### 注意事项

✅ **推荐**:
- 计数器、浏览量、点赞数
- 库存扣减（配合事务）
- 余额、积分变更

❌ **不推荐**:
- 需要记录变更历史的场景
- 需要业务逻辑校验的场景

## 乐观锁

### 功能说明

乐观锁通过版本号机制实现，每次更新时检查版本号是否变化，适合**读多写少**的低冲突场景。

### 数据库要求

需要在表中添加 `version` 字段：

```sql
-- MySQL
ALTER TABLE products ADD COLUMN version INT DEFAULT 1;

-- 或者创建表时
CREATE TABLE products (
    id INT PRIMARY KEY,
    name VARCHAR(100),
    stock INT,
    version INT DEFAULT 1  -- 乐观锁版本字段
);
```

### API 接口

```go
// 启用乐观锁
WithVersionLock(field ...string) *Model

// 默认版本字段："version"
// 自定义版本字段：WithVersionLock("ver")
```

### 使用示例

#### 1. 基础用法

```go
// 1. 查询数据（带版本号）
product := model.New("products").
    WithVersionLock().
    Find(1001)

// 2. 修改数据
product.SetAttr("price", 99.99)
product.SetAttr("stock", product.GetAttr("stock").(int) - 1)

// 3. 保存（自动检查版本号）
_, err := product.Save()

if err != nil {
    if strings.Contains(err.Error(), "optimistic lock failed") {
        log.Println("数据已被其他进程修改，请重试")
    }
    return err
}
```

#### 2. 自定义版本字段

```go
// 使用 custom_version 作为版本字段
model := model.New("products").
    WithVersionLock("custom_version").
    Find(1001)

model.SetAttr("price", 99.99)
_, err := model.Save()
```

#### 3. 事务中使用

```go
err := db.Transaction(func(q *db.Query) error {
    // 1. 查询商品
    product := model.New("products").
        WithVersionLock().
        Find(1001)
    
    stock := product.GetAttr("stock").(int)
    if stock < 1 {
        return fmt.Errorf("库存不足")
    }
    
    // 2. 扣减库存
    product.SetAttr("stock", stock - 1)
    _, err := product.Save()
    if err != nil {
        return err
    }
    
    // 3. 创建订单
    _, err = q.Table("orders").Insert(orderData)
    return err
})
```

#### 4. 重试机制

```go
func updateWithRetry(model *model.Model, data map[string]interface{}, maxRetries int) error {
    for i := 0; i < maxRetries; i++ {
        // 重新查询
        m := model.New(model.GetTable()).
            WithVersionLock().
            Find(model.GetAttr("id"))
        
        // 设置数据
        for k, v := range data {
            m.SetAttr(k, v)
        }
        
        // 保存
        _, err := m.Save()
        if err == nil {
            return nil // 成功
        }
        
        if !strings.Contains(err.Error(), "optimistic lock failed") {
            return err // 其他错误，直接返回
        }
        
        // 版本冲突，重试
        log.Printf("版本冲突，第 %d 次重试", i+1)
        time.Sleep(time.Duration(i*10) * time.Millisecond) // 退避
    }
    
    return fmt.Errorf("optimistic lock retry failed")
}
```

### 性能指标

| 场景 | 冲突率 | QPS | 延迟 |
|------|--------|-----|------|
| **低冲突** (<10%) | 5% | 3,000+ | 1-2ms |
| **中冲突** (10-30%) | 20% | 1,500+ | 2-5ms |
| **高冲突** (>30%) | 50% | 500+ | 5-10ms |

### 注意事项

✅ **适合场景**:
- 配置修改（系统配置、用户设置）
- 订单状态变更（低并发）
- 文章、商品信息发布

❌ **不适合场景**:
- 高并发写入（如秒杀）
- 频繁修改的计数器
- 库存扣减（建议用原子操作）

⚠️ **重要**:
- 必须设置 version 字段默认值为 1
- 需要配合重试机制处理冲突
- 高冲突场景性能差

## 分布式锁

### 功能说明

分布式锁基于 Redis 实现，用于**多实例部署**时的进程间同步，防止重复执行。

### 环境要求

- Redis 2.6+（支持 Lua 脚本）
- 推荐 Redis 4.0+（支持更多命令）

### API 接口

```go
// 基础分布式锁
NewRedisLock(client *redis.Client, key string, config *RedisLockConfig) *RedisLock
Lock(ctx context.Context) bool
Unlock(ctx context.Context) error

// 看门狗自动续期
WithWatchdogLock(ctx context.Context, lock Lock, fn func() error) error

// RedLock 多实例锁
NewRedLock(clients []*redis.Client, key string, config *RedLockConfig) *RedLock
Lock(ctx context.Context) bool
Unlock(ctx context.Context) error
```

### 配置参数

```go
type RedisLockConfig struct {
    TTL      time.Duration  // 锁超时时间（默认 30 秒）
    Retry    int            // 重试次数（默认 3）
    Interval time.Duration  // 重试间隔（默认 100ms）
}
```

### 使用示例

#### 1. 基础用法

```go
import "vigo/framework/lock"

// 创建锁
lock := lock.NewRedisLock(redisClient, "product:1001:stock")

// 尝试获取锁
if lock.Lock(ctx) {
    defer lock.Unlock(ctx)
    
    // 执行业务逻辑
    err := processOrder()
    if err != nil {
        return err
    }
} else {
    return fmt.Errorf("获取锁失败，操作正在进行中")
}
```

#### 2. 便捷函数

```go
// 自动获取和释放锁
err := lock.WithLock(ctx, redisClient, "order:1001", func() error {
    return processOrder()
})

if err != nil {
    log.Printf("处理失败：%v", err)
}
```

#### 3. 看门狗自动续期（长任务）

```go
// 长任务（可能超过 TTL）
lock := lock.NewRedisLock(redisClient, "task:long_running")

if lock.Lock(ctx) {
    // 使用看门狗自动续期
    err := lock.WithWatchdogLock(ctx, lock, func() error {
        // 执行长任务（自动续期）
        return longRunningTask() // 可能执行 5 分钟
    })
    
    if err != nil {
        log.Printf("任务失败：%v", err)
    }
}
```

#### 4. 防止定时任务重复执行

```go
func scheduledTask(ctx context.Context) {
    lock := lock.NewRedisLock(redisClient, "cron:daily_report")
    
    if !lock.Lock(ctx) {
        log.Println("任务正在其他实例运行，跳过")
        return
    }
    defer lock.Unlock(ctx)
    
    // 生成日报
    generateDailyReport()
}
```

#### 5. RedLock 多实例锁（高可用）

```go
// 5 个 Redis 实例（推荐奇数个）
clients := []*redis.Client{
    redisClient1, // Redis 1
    redisClient2, // Redis 2
    redisClient3, // Redis 3
    redisClient4, // Redis 4
    redisClient5, // Redis 5
}

// 创建 RedLock
redLock := lock.NewRedLock(clients, "critical:lock")

if redLock.Lock(ctx) {
    defer redLock.Unlock(ctx)
    
    // 需要获得 3 个以上实例的锁才能执行
    err := criticalOperation()
    if err != nil {
        return err
    }
} else {
    return fmt.Errorf("无法获得多数实例的锁")
}
```

#### 6. 秒杀场景（完整示例）

```go
func seckill(ctx context.Context, userID, productID int) error {
    // 1. 分布式锁（防止超卖）
    lockKey := fmt.Sprintf("seckill:product:%d", productID)
    lock := lock.NewRedisLock(redisClient, lockKey, &lock.RedisLockConfig{
        TTL:      10 * time.Second,
        Retry:    3,
        Interval: 100 * time.Millisecond,
    })
    
    if !lock.Lock(ctx) {
        return fmt.Errorf("系统繁忙，请稍后再试")
    }
    defer lock.Unlock(ctx)
    
    // 2. 数据库事务 + 原子操作
    err := db.Transaction(func(q *db.Query) error {
        // 检查库存
        product := model.New("products").Find(productID)
        stock := product.GetAttr("stock").(int)
        
        if stock < 1 {
            return fmt.Errorf("库存已售罄")
        }
        
        // 原子扣减库存
        _, err := q.Table("products").
            Where("id", productID).
            Decrement("stock", 1)
        
        if err != nil {
            return err
        }
        
        // 创建订单
        _, err = q.Table("orders").Insert(map[string]interface{}{
            "user_id":     userID,
            "product_id":  productID,
            "status":      "pending",
            "created_at":  time.Now(),
        })
        
        return err
    })
    
    if err != nil {
        return err
    }
    
    // 3. 异步处理（扣款、发货等）
    go processOrderAsync(userID, productID)
    
    return nil
}
```

### 性能指标

| 操作 | 延迟 | QPS | 说明 |
|------|------|-----|------|
| Lock (无竞争) | 1ms | 1,000+ | 单次 SetNX |
| Lock (有竞争) | 3-10ms | 300+ | 包含重试 |
| Unlock | 0.5ms | 2,000+ | Lua 脚本 |
| RedLock | 5-20ms | 100+ | 需要多数实例 |

### 注意事项

✅ **推荐场景**:
- 多实例部署的任务同步
- 定时任务防重复
- 秒杀、抢购
- 关键业务操作互斥

❌ **不推荐场景**:
- 单机应用（用原子操作/乐观锁即可）
- 高频计数器（性能开销大）
- 读操作（不需要锁）

⚠️ **重要**:
- **必须设置 TTL** - 防止死锁
- **看门狗用于长任务** - 自动续期
- **RedLock 需要奇数个实例** - 至少 3 个
- **解锁用 Lua 脚本** - 保证原子性

## 最佳实践

### 1. 选型原则

```
优先顺序：原子操作 > 乐观锁 > 分布式锁

1. 能用原子操作的，不用锁
2. 读多写少的，用乐观锁
3. 多实例同步的，用分布式锁
4. 高并发写入的，用原子操作 + 分布式锁
```

### 2. 性能优化

```go
// ✅ 好的做法：原子操作
db.Table("products").
    Where("id", 1001).
    Decrement("stock", 1)

// ❌ 差的做法：先查后改（有竞态条件）
product := model.Find(1001)
stock := product.GetAttr("stock").(int)
if stock < 1 {
    return errors.New("库存不足")
}
product.SetAttr("stock", stock - 1)
product.Save()  // 可能超卖
```

### 3. 事务配合

```go
// ✅ 推荐：事务 + 原子操作
err := db.Transaction(func(q *db.Query) error {
    // 原子扣减
    _, err := q.Table("products").
        Where("id", productID).
        Decrement("stock", 1)
    
    if err != nil {
        return err
    }
    
    // 创建订单
    _, err = q.Table("orders").Insert(orderData)
    return err
})
```

### 4. 重试机制

```go
// 乐观锁重试
func updateWithRetry(fn func() error, maxRetries int) error {
    for i := 0; i < maxRetries; i++ {
        if err := fn(); err == nil {
            return nil
        }
        time.Sleep(time.Duration(i*10) * time.Millisecond)
    }
    return fmt.Errorf("retry failed")
}

// 分布式锁重试
config := &lock.RedisLockConfig{
    Retry:    3,
    Interval: 100 * time.Millisecond,
    TTL:      30 * time.Second,
}
```

### 5. 监控告警

```go
// 监控锁等待时间
start := time.Now()
if !lock.Lock(ctx) {
    metrics.LockFail.Inc()
    return errors.New("获取锁失败")
}
defer lock.Unlock(ctx)

duration := time.Since(start)
if duration > time.Second {
    log.Printf("锁等待时间过长：%v", duration)
    metrics.LockWaitTime.Observe(duration.Seconds())
}
```

## 常见问题

### Q1: 原子操作和事务有什么区别？

**A**: 
- **原子操作**: 单条 SQL，数据库层面保证原子性，性能最优
- **事务**: 多条 SQL 要么全部成功，要么全部失败，可回滚

**建议**: 单表操作用原子操作，多表操作用事务 + 原子操作。

---

### Q2: 乐观锁冲突了怎么办？

**A**: 
1. **重试机制** - 自动重试 3-5 次
2. **退避策略** - 每次重试间隔递增
3. **提示用户** - 告知数据已变更，请刷新后重试

```go
// 重试示例
for i := 0; i < 3; i++ {
    model := model.New("products").
        WithVersionLock().
        Find(1001)
    
    model.SetAttr("price", 99.99)
    _, err := model.Save()
    
    if err == nil {
        return nil // 成功
    }
    
    if !strings.Contains(err.Error(), "optimistic lock") {
        return err // 其他错误
    }
    
    time.Sleep(time.Duration(i*100) * time.Millisecond)
}
```

---

### Q3: 分布式锁 TTL 设置多少合适？

**A**: 
- **短任务** (< 1 秒): TTL = 5-10 秒
- **中等任务** (1-10 秒): TTL = 30 秒
- **长任务** (> 10 秒): TTL = 30 秒 + 看门狗

**原则**: TTL 略大于任务执行时间，但不宜过长（防止死锁）。

---

### Q4: 看门狗的原理是什么？

**A**: 
看门狗通过后台 goroutine 定期续期锁的 TTL：

```go
// 简化原理
go func() {
    ticker := time.NewTicker(TTL / 3)
    defer ticker.Stop()
    
    for {
        select {
        case <-ticker.C:
            // 续期
            client.Expire(key, TTL)
        case <-done:
            // 任务完成，停止续期
            return
        }
    }
}()
```

---

### Q5: RedLock 和单 Redis 锁有什么区别？

**A**: 

| 特性 | 单 Redis 锁 | RedLock |
|------|------------|---------|
| **可用性** | 单点故障 | 高可用 |
| **性能** | 高（1 次请求） | 低（N 次请求） |
| **一致性** | 弱 | 强 |
| **适用** | 一般场景 | 关键业务 |

**建议**: 一般场景用单 Redis 锁，关键业务用 RedLock。

---

### Q6: 如何调试锁问题？

**A**: 
1. **日志记录** - 记录加锁/解锁时间
2. **监控指标** - 监控锁等待时间、失败次数
3. **超时检测** - 设置合理的 TTL 和超时告警

```go
// 调试日志
log.Printf("尝试获取锁：%s", lockKey)
if !lock.Lock(ctx) {
    log.Printf("获取锁失败：%s", lockKey)
    return
}
defer func() {
    log.Printf("释放锁：%s", lockKey)
    lock.Unlock(ctx)
}()
```

## 总结

### 选择指南

```
1. 计数器、库存 → 原子操作
2. 配置、状态 → 乐观锁
3. 多实例同步 → 分布式锁
4. 高并发 + 多实例 → 原子操作 + 分布式锁
5. 长任务 → 分布式锁 + 看门狗
```

### 性能对比

| 锁类型 | QPS | 延迟 | 适用场景 |
|--------|-----|------|----------|
| 原子操作 | 5,000+ | 0.5-1ms | 计数器、库存 |
| 乐观锁 | 3,000+ | 1-2ms | 配置、状态 |
| 分布式锁 | 1,000+ | 1-10ms | 多实例同步 |
| RedLock | 100+ | 5-20ms | 关键业务 |

### 核心原则

1. **能不用锁就不用锁** - 原子操作优先
2. **锁的粒度要小** - 只锁必要的代码段
3. **锁的时间要短** - 快速执行，快速释放
4. **必须有超时** - 防止死锁
5. **监控和告警** - 及时发现问题
