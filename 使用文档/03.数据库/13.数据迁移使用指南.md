# 🔄 数据迁移使用指南

## 简介

Vigo 框架提供了完整的数据库迁移（Migration）功能，用于管理数据库 schema 的版本控制和变更。

### 主要特性

- ✅ **版本控制**：每个迁移都有唯一的版本号
- ✅ **向上/向下迁移**：支持应用变更（Up）和回滚变更（Down）
- ✅ **自动记录**：自动记录已应用的迁移
- ✅ **批量操作**：支持批量执行、回滚迁移
- ✅ **状态查询**：随时查看迁移执行状态
- ✅ **CLI 支持**：支持命令行和 API 两种方式

## 快速开始

### 1. 创建迁移文件

迁移文件位于 `database/migrations/` 目录下，命名格式为：

```
{version}_{name}.go
```

**示例：**
```bash
database/migrations/
├── 20260307120000_create_users_table.go
├── 20260307120100_create_articles_table.go
└── 20260307120200_create_comments_table.go
```

**版本号规则：**
- 使用时间戳格式：`YYYYMMDDHHMMSS`
- 例如：`20260307120000` 表示 2026-03-07 12:00:00

### 2. 编写迁移文件

每个迁移文件包含两个函数：`Up` 和 `Down`

**示例：创建用户表**

```go
package migrations

import (
	"database/sql"
	"log"
)

// Up 创建用户表（向上迁移）
func Up_20260307120000_create_users_table(db *sql.DB) error {
	log.Println("Creating users table...")
	
	query := `
		CREATE TABLE IF NOT EXISTS users (
			id BIGINT AUTO_INCREMENT PRIMARY KEY,
			username VARCHAR(50) NOT NULL UNIQUE,
			email VARCHAR(100) NOT NULL UNIQUE,
			password VARCHAR(255) NOT NULL,
			role VARCHAR(20) DEFAULT 'user',
			status TINYINT DEFAULT 1,
			created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
			updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
			deleted_at TIMESTAMP NULL,
			INDEX idx_username (username),
			INDEX idx_email (email),
			INDEX idx_status (status)
		) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
	`
	
	_, err := db.Exec(query)
	if err != nil {
		return err
	}
	
	log.Println("Users table created successfully")
	return nil
}

// Down 删除用户表（向下迁移/回滚）
func Down_20260307120000_create_users_table(db *sql.DB) error {
	log.Println("Dropping users table...")
	
	_, err := db.Exec("DROP TABLE IF EXISTS users")
	if err != nil {
		return err
	}
	
	log.Println("Users table dropped successfully")
	return nil
}
```

### 3. 注册迁移函数

在应用启动时注册迁移函数：

```go
package main

import (
	"vigo/framework/db"
	"vigo/database/migrations"
)

func init() {
	// 注册迁移
	migrator := db.NewMigrator(db.GlobalDB, "migrations")
	
	migrator.AddMigration(
		20260307120000,
		"create_users_table",
		migrations.Up_20260307120000_create_users_table,
		migrations.Down_20260307120000_create_users_table,
	)
	
	migrator.AddMigration(
		20260307120100,
		"create_articles_table",
		migrations.Up_20260307120100_create_articles_table,
		migrations.Down_20260307120100_create_articles_table,
	)
}

func main() {
	// 自动执行迁移
	migrator := db.NewMigrator(db.GlobalDB, "migrations")
	migrator.Migrate()
	
	// ... 其他初始化代码
}
```

## 使用方式

### 方式一：API 接口

框架提供了完整的 HTTP API 来管理迁移。

#### 1. 查看迁移状态

```bash
GET /api/migration/status
```

**响应示例：**
```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "current_version": 20260307120000,
    "total": 2,
    "applied": [
      {
        "version": 20260307120000,
        "name": "create_users_table",
        "applied_at": "2026-03-07T12:00:00Z"
      }
    ],
    "pending": [
      {
        "version": 20260307120100,
        "name": "create_articles_table"
      }
    ]
  }
}
```

#### 2. 执行迁移

```bash
POST /api/migration/migrate
```

**响应示例：**
```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "message": "迁移执行成功"
  }
}
```

#### 3. 回滚迁移

```bash
POST /api/migration/rollback?steps=1
```

**参数：**
- `steps` (可选): 回滚步数，默认 1

**响应示例：**
```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "message": "成功回滚 1 步迁移"
  }
}
```

#### 4. 重置所有迁移

```bash
POST /api/migration/reset
```

**警告：** 此操作会删除所有迁移记录并回滚所有迁移！

**响应示例：**
```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "message": "所有迁移已重置"
  }
}
```

### 方式二：命令行

通过框架 CLI 工具执行迁移：

```bash
# 执行所有未应用的迁移
./vigo migrate

# 回滚 1 步迁移
./vigo rollback

# 回滚 3 步迁移
./vigo rollback 3

# 重置所有迁移
./vigo reset

# 查看迁移状态
./vigo status
```

### 方式三：代码调用

在 Go 代码中直接调用：

```go
package main

import (
	"log"
	"vigo/framework/db"
)

func main() {
	// 创建迁移管理器
	migrator := db.NewMigrator(db.GlobalDB, "migrations")
	
	// 从目录加载迁移文件
	err := migrator.LoadMigrationsFromDir("database/migrations")
	if err != nil {
		log.Fatal(err)
	}
	
	// 执行迁移
	err = migrator.Migrate()
	if err != nil {
		log.Fatal(err)
	}
	
	// 查看状态
	applied, pending, err := migrator.Status()
	if err != nil {
		log.Fatal(err)
	}
	
	log.Printf("Applied: %d, Pending: %d\n", len(applied), len(pending))
	
	// 回滚最后 1 步
	err = migrator.Rollback(1)
	if err != nil {
		log.Fatal(err)
	}
	
	// 获取当前版本
	version, _ := migrator.GetCurrentVersion()
	log.Printf("Current version: %d\n", version)
}
```

## 最佳实践

### 1. 迁移文件命名

- 使用有意义的名词描述变更内容
- 使用时间戳作为版本号，确保唯一性
- 遵循统一的命名规范

**好的示例：**
```
20260307120000_create_users_table.go
20260307120100_add_email_to_users.go
20260307120200_create_posts_and_comments.go
```

**不好的示例：**
```
001_create_table.go
002_add_column.go
test_migration.go
```

### 2. 编写迁移

- **Up 迁移**：只包含必要的变更操作
- **Down 迁移**：必须能够完全撤销 Up 迁移的操作
- **幂等性**：迁移应该是幂等的，可以安全地重复执行
- **事务安全**：复杂迁移应该使用事务

**示例：使用事务**
```go
func Up_add_user_profile(db *sql.DB) error {
	tx, err := db.Begin()
	if err != nil {
		return err
	}
	defer tx.Rollback()
	
	// 添加字段
	_, err = tx.Exec(`
		ALTER TABLE users 
		ADD COLUMN avatar VARCHAR(255),
		ADD COLUMN bio TEXT,
		ADD COLUMN birthday DATE
	`)
	if err != nil {
		return err
	}
	
	// 创建索引
	_, err = tx.Exec(`
		CREATE INDEX idx_users_birthday ON users(birthday)
	`)
	if err != nil {
		return err
	}
	
	return tx.Commit()
}
```

### 3. 数据迁移

- **不要修改已发布的迁移**：已应用的迁移不应该被修改
- **新增而非修改**：如果需要修改，创建新的迁移来撤销并重新创建
- **数据备份**：在执行数据变更前备份重要数据
- **测试环境验证**：先在生产环境验证迁移

**示例：添加数据**
```go
func Up_add_default_roles(db *sql.DB) error {
	_, err := db.Exec(`
		INSERT INTO roles (name, description, created_at) 
		VALUES 
			('admin', '系统管理员', NOW()),
			('editor', '内容编辑', NOW()),
			('user', '普通用户', NOW())
		ON DUPLICATE KEY UPDATE name=name
	`)
	return err
}

func Down_remove_default_roles(db *sql.DB) error {
	_, err := db.Exec(`
		DELETE FROM roles 
		WHERE name IN ('admin', 'editor', 'user')
	`)
	return err
}
```

### 4. 版本控制

- 将迁移文件纳入版本控制（Git）
- 确保团队成员都能获取迁移历史
- 在代码审查时检查迁移文件

### 5. 生产环境部署

- **自动化部署**：在 CI/CD 流程中自动执行迁移
- **回滚计划**：始终准备好回滚方案
- **监控日志**：记录迁移执行过程
- **灰度发布**：先在部分节点执行迁移

## 常见问题

### Q1: 迁移执行失败怎么办？

**A:** 
1. 检查错误日志，定位失败原因
2. 修复问题后，重新执行 `migrate`
3. 如果需要回滚，使用 `rollback` 命令

### Q2: 可以跳过某个迁移吗？

**A:** 
不建议跳过迁移。迁移应该按顺序执行，确保数据库结构的一致性。

### Q3: 如何修改已执行的迁移？

**A:** 
1. **未发布**：如果迁移还未发布到生产，可以直接修改并重置迁移
2. **已发布**：创建新的迁移来撤销并重新创建

**示例：修改已发布的迁移**
```go
// 原迁移：添加 email 字段
func Up_add_email(db *sql.DB) error {
	_, err := db.Exec("ALTER TABLE users ADD COLUMN email VARCHAR(100)")
	return err
}

// 新迁移：修改 email 字段类型
func Up_change_email_type(db *sql.DB) error {
	_, err := db.Exec("ALTER TABLE users MODIFY COLUMN email VARCHAR(255)")
	return err
}
```

### Q4: 迁移表是什么？

**A:** 
迁移表（默认名：`migrations`）用于记录已执行的迁移：

```sql
CREATE TABLE migrations (
	id BIGINT AUTO_INCREMENT PRIMARY KEY,
	version BIGINT NOT NULL UNIQUE,
	name VARCHAR(255) NOT NULL,
	applied_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### Q5: 如何在测试中使用迁移？

**A:** 
在测试前执行迁移，测试后回滚：

```go
func TestMain(m *testing.M) {
	// 执行迁移
	migrator := db.NewMigrator(testDB, "migrations")
	migrator.Migrate()
	
	// 运行测试
	code := m.Run()
	
	// 回滚迁移
	migrator.Rollback(999) // 回滚所有
	
	os.Exit(code)
}
```
