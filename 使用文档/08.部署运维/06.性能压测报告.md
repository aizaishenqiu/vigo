# Vigo 框架性能压测总结报告

## 一、测试概述

本次压测针对 Vigo 框架的核心能力，分别在 **500 并发** 和 **1000 并发** 下，对 **HTTP 接口层** 与 **中间件层（MySQL + Redis）** 进行性能验证，评估框架在不同负载下的吞吐能力、延迟表现及稳定性。

---

|---------|--------|---------|---------|---------|
| HTTP 接口压测 | 500 | 10s | 健康检查接口（GET） | 目标 URL：http://127.0.0.1:8080/health |

## 四、性能表现分析

### 1. 中间件层（MySQL + Redis）

- **吞吐能力**：在 1000 并发下 QPS 达到 50,000，相比 500 并发实现线性翻倍，展现出极强的扩展能力。
- **延迟表现**：平均延迟从 23.30ms 增加到 46.47ms，但 P95/P99 延迟与平均延迟接近，无明显长尾效应，性能稳定可控。
- **稳定性**：全程零失败请求，是整个架构中最可靠的性能基石。

### 2. HTTP 接口层

- **吞吐能力**：从 500 并发到 1000 并发，QPS 仅从 17,420 提升至 19,180，增幅有限，说明应用层处理能力接近饱和。
- **延迟表现**：平均延迟从 26.67ms 飙升至 51.70ms，P99 延迟更是达到 88.52ms，长尾延迟显著恶化。
- **稳定性**：在 1000 并发后期出现大量失败请求，表明 HTTP 接口层已成为性能瓶颈。

### 3. 资源利用率

- **CPU**：中间件层 CPU 使用率随并发线性增长，而 HTTP 层 CPU 使用率在 500 并发后趋于平稳，说明瓶颈不在硬件，而在应用层处理逻辑。
- **内存**：两种场景下内存均稳定在～25GB，无泄漏或激增，资源使用健康。

## 五、结论与优化建议

### 结论

1. **中间件层表现优异**：MySQL + Redis 读写性能强劲，可线性支撑高并发，是架构的核心优势。
2. **HTTP 应用层存在瓶颈**：在 1000 并发下延迟翻倍且出现请求失败，是当前性能短板。
3. **整体稳定性良好**：500 并发下全场景无失败，1000 并发下中间件层仍保持 100% 成功率，框架基础可靠。

### 优化建议

#### 应用层优化

1. **检查健康检查接口逻辑**，移除不必要的计算或 IO 操作
2. **对接口结果进行缓存**，或采用异步处理提升吞吐
3. **优化 JSON 序列化**，使用更高效的编码器

#### 水平扩展

- 对 HTTP 服务进行多实例部署，通过负载均衡分散压力，突破单机性能限制

#### 持续验证

- 优化后再次进行 1000/2000 并发压测，验证性能提升效果

## 六、已实施的优化措施

基于本次压测结果，已对框架进行以下优化：

### 1. 健康检查接口优化

**问题**：原 `/health` 接口每次请求都执行 DB 和 Redis 的 Ping 操作，高并发下成为性能瓶颈。

**优化方案**：
- 拆分为两个接口：
  - `/health` - 轻量级健康检查，仅返回服务状态，无 IO 操作
  - `/health/full` - 完整健康检查，包含 DB 和 Redis 状态，带 3 秒缓存

```go
// 轻量级健康检查 - 无 IO 操作
r.GET("/health", healthCheck)

// 完整健康检查 - 带 3 秒缓存
r.GET("/health/full", healthCheckFull)
```

### 2. JSON 序列化优化

**问题**：原 `Json()` 方法直接写入 ResponseWriter，无法复用内存。

**优化方案**：
- 使用 `sync.Pool` 复用 `bytes.Buffer`
- 预先编码 JSON 到缓冲区，再一次性写入

```go
var bufferPool = sync.Pool{
    New: func() interface{} {
        return new(bytes.Buffer)
    },
}

func (c *Context) Json(code int, data interface{}) {
    buf := bufferPool.Get().(*bytes.Buffer)
    buf.Reset()
    defer bufferPool.Put(buf)
    // ... 编码并写入
}
```

### 3. 响应缓存中间件

**新增**：`ResponseCacheMiddleware` 中间件，支持 GET 请求响应缓存。

```go
// 使用示例：5 秒缓存
r.Use(middleware.ResponseCacheMiddleware(5 * time.Second))
```

### 4. 连接池配置优化

配置文件已针对 10 万并发进行优化：

```yaml
database:
  max_open_conns: 1000    # 最大打开连接数
  max_idle_conns: 500     # 最大空闲连接数
  conn_max_lifetime: 1800 # 连接最大生命周期（秒）
  conn_max_idletime: 300  # 空闲连接回收时间（秒）

redis:
  pool_size: 2000         # 最大连接池大小
  min_idle_conns: 200     # 最小空闲连接数
  max_idle_conns: 1000    # 最大空闲连接数
```
