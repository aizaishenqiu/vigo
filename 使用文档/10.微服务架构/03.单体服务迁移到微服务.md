# 单体服务迁移到微服务

## 简介
当单体服务发展到一定规模，面临性能瓶颈、团队协作困难等问题时，需要考虑迁移到微服务架构。本文档介绍如何平稳地将单体服务迁移到微服务。

## 迁移前评估

### 是否需要迁移
```
需要迁移的情况：
1. 代码库过大，编译部署时间过长
2. 团队成员多，代码冲突频繁
3. 部分模块需要独立扩展
4. 不同模块有不同的技术栈需求
5. 故障隔离需求高

暂不需要迁移的情况：
1. 团队规模小（<10人）
2. 业务相对简单
3. 没有明显的性能瓶颈
4. 运维能力不足
```

### 迁移风险评估
```
风险点：
1. 数据一致性：跨服务事务处理
2. 服务通信：网络延迟、超时处理
3. 运维复杂度：服务数量增加
4. 学习成本：团队需要掌握新技术
5. 调试困难：分布式系统问题定位
```

## 迁移策略

### 渐进式迁移
```
单体服务 → 模块化单体 → 微服务

步骤：
1. 模块化改造：在单体内部划分清晰模块
2. 接口抽象：模块间通过接口通信
3. 独立数据库：每个模块使用独立数据库
4. 逐步拆分：按优先级逐个拆分服务
```

#### 模块接口定义
```go
// modules/module.go
package modules

import "net/http"

// Module 模块接口
type Module interface {
    Name() string
    RegisterRoutes(mux *http.ServeMux)
    Init() error
}

// ModuleManager 模块管理器
type ModuleManager struct {
    modules []Module
}

func NewModuleManager() *ModuleManager {
    return &ModuleManager{
        modules: make([]Module, 0),
    }
}

func (mm *ModuleManager) Register(m Module) {
    mm.modules = append(mm.modules, m)
}

func (mm *ModuleManager) InitAll() error {
    for _, m := range mm.modules {
        if err := m.Init(); err != nil {
            return fmt.Errorf("模块 %s 初始化失败: %v", m.Name(), err)
        }
    }
    return nil
}

func (mm *ModuleManager) RegisterAllRoutes(mux *http.ServeMux) {
    for _, m := range mm.modules {
        m.RegisterRoutes(mux)
    }
}
```

```go
// core/db/manager.go
package db

import (
    "database/sql"
    "sync"
)

var (
    instances = make(map[string]*sql.DB)
    mu        sync.RWMutex
)

// InitDB 初始化指定数据库
func InitDB(name, dsn string, maxOpen, maxIdle int) error {
    db, err := sql.Open("mysql", dsn)
    if err != nil {
        return err
    }
    
    db.SetMaxOpenConns(maxOpen)
    db.SetMaxIdleConns(maxIdle)
    
    mu.Lock()
    instances[name] = db
    mu.Unlock()
    
    return nil
}

// GetDB 获取指定数据库连接
func GetDB(name string) *sql.DB {
    mu.RLock()
    defer mu.RUnlock()
    return instances[name]
}
```

#### 通过API访问其他模块数据
```go
// modules/user/client.go
package user

import (
    "encoding/json"
    "net/http"
)

// UserClient 用户模块客户端
type UserClient struct {
    baseURL string
}

func NewUserClient(baseURL string) *UserClient {
    return &UserClient{baseURL: baseURL}
}

func (uc *UserClient) GetUserByID(id int64) (*User, error) {
    resp, err := http.Get(fmt.Sprintf("%s/internal/users/%d", uc.baseURL, id))
    if err != nil {
        return nil, err
    }
    defer resp.Body.Close()
    
    var result struct {
        Code int   `json:"code"`
        Data *User `json:"data"`
    }
    
    if err := json.NewDecoder(resp.Body).Decode(&result); err != nil {
        return nil, err
    }
    
    return result.Data, nil
}
```

### 第三步：拆分服务

#### 拆分用户服务
```go
// user-service/main.go
package main

import (
    "net/http"
    
    "user-service/controller"
    "user-service/model"
    "user-service/db"
)

func main() {
    // 初始化用户数据库
    db.Init("mysql", "root:pass@tcp(user-db:3306)/user_db", 50, 5, 3600)
    
    mux := http.NewServeMux()
    
    // 外部API
    mux.HandleFunc("/api/users", controller.UserList)
    mux.HandleFunc("/api/users/", controller.UserDetail)
    mux.HandleFunc("/api/auth/login", controller.Login)
    
    // 内部API（供其他服务调用）
    mux.HandleFunc("/internal/users/", controller.InternalUserDetail)
    
    http.ListenAndServe(":8001", mux)
}
```

```go
// user-service/controller/internal.go
package controller

import (
    "encoding/json"
    "net/http"
    "strconv"
    
    "user-service/model"
)

// InternalUserDetail 内部API，供其他服务调用
func InternalUserDetail(w http.ResponseWriter, r *http.Request) {
    idStr := r.URL.Path[len("/internal/users/"):]
    id, err := strconv.ParseInt(idStr, 10, 64)
    if err != nil {
        http.Error(w, "无效的用户ID", http.StatusBadRequest)
        return
    }
    
    user := &model.User{}
    userInfo, err := user.FindByID(id)
    if err != nil {
        http.Error(w, err.Error(), http.StatusInternalServerError)
        return
    }
    
    w.Header().Set("Content-Type", "application/json")
    json.NewEncoder(w).Encode(map[string]interface{}{
        "code": 0,
        "data": userInfo,
    })
}
```

### 第四步：API网关

```go
// gateway/main.go
package main

import (
    "net/http"
    "net/http/httputil"
    "net/url"
    "strings"
)

func main() {
    routes := map[string]string{
        "/api/users":    "http://user-service:8001",
        "/api/auth":     "http://user-service:8001",
        "/api/orders":   "http://order-service:8002",
        "/api/products": "http://product-service:8003",
    }
    
    handler := func(w http.ResponseWriter, r *http.Request) {
        for prefix, target := range routes {
            if strings.HasPrefix(r.URL.Path, prefix) {
                proxy(w, r, target)
                return
            }
        }
        http.NotFound(w, r)
    }
    
    http.ListenAndServe(":8080", http.HandlerFunc(handler))
}

func proxy(w http.ResponseWriter, r *http.Request, target string) {
    targetURL, _ := url.Parse(target)
    
    proxy := httputil.NewSingleHostReverseProxy(targetURL)
    proxy.Director = func(req *http.Request) {
        req.URL.Scheme = targetURL.Scheme
        req.URL.Host = targetURL.Host
        req.Header.Set("X-Forwarded-Host", r.Host)
    }
    
    proxy.ServeHTTP(w, r)
}
```

## 数据迁移

### 数据拆分策略
```sql
-- 原始数据库：myapp_db
-- 包含所有表：users, orders, products, payments...

-- 1. 创建用户数据库
CREATE DATABASE user_db;
-- 迁移用户相关表
CREATE TABLE user_db.users LIKE myapp_db.users;
INSERT INTO user_db.users SELECT * FROM myapp_db.users;

-- 2. 创建订单数据库
CREATE DATABASE order_db;
-- 迁移订单相关表
CREATE TABLE order_db.orders LIKE myapp_db.orders;
INSERT INTO order_db.orders SELECT * FROM myapp_db.orders;

-- 3. 创建商品数据库
CREATE DATABASE product_db;
-- 迁移商品相关表
CREATE TABLE product_db.products LIKE myapp_db.products;
INSERT INTO product_db.products SELECT * FROM myapp_db.products;
```

## 迁移检查清单

### 迁移前检查
```
□ 评估迁移必要性和风险
□ 确定拆分边界和服务数量
□ 制定详细的迁移计划
□ 准备基础设施（服务器、数据库、消息队列等）
□ 团队技术培训
```

### 迁移中检查
```
□ 模块化改造完成
□ 数据库分离完成
□ 服务间通信测试通过
□ API网关配置完成
□ 监控和日志系统就绪
```

### 迁移后检查
```
□ 功能测试通过
□ 性能测试达标
□ 数据一致性验证
□ 故障恢复测试
□ 文档更新完成
```

## 注意事项
1. 渐进式迁移，不要一次性拆分所有服务
2. 保持向后兼容，逐步切换流量
3. 做好数据备份和回滚方案
4. 监控系统要先行，确保可观测性
5. 服务拆分后要做好服务治理
6. 文档和团队沟通要及时跟上
7. 预留足够的迁移时间和资源
