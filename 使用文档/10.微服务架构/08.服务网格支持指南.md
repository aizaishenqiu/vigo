# 服务网格支持

Vigo 框架提供服务网格（Service Mesh）集成能力，支持 Istio、Linkerd 等主流服务网格方案，帮助开发者构建更加灵活、可扩展的微服务架构。

## 概述

服务网格是基础设施层，用于处理服务间的通信，提供服务发现、负载均衡、熔断限流、链路追踪、安全认证等能力，而无需修改业务代码。

### Vigo 的服务网格支持

- ✅ **透明代理**: 通过 Sidecar 模式实现服务间通信
- ✅ **流量管理**: 支持流量拆分、金丝雀发布、蓝绿部署
- ✅ **可观测性**: 集成链路追踪、指标监控、日志收集
- ✅ **安全通信**: mTLS 加密、身份认证、访问控制
- ✅ **弹性能力**: 熔断、降级、重试、超时控制

## 架构设计

### Sidecar 模式

```
┌─────────────────┐     ┌─────────────────┐
│   Application   │     │   Application   │
│   (Vigo App)    │     │   (Vigo App)    │
└────────┬────────┘     └────────┬────────┘
         │                       │
         ├───────────────────────┤
         │    Sidecar Proxy      │
         │  (Envoy / Linkerd)    │
         └───────────────────────┘
                 │
         ┌───────┴───────┐
         │  Service Mesh │
         │   Control     │
         │    Plane      │
         └───────────────┘
```

### 集成方式

Vigo 应用作为数据平面的一部分，通过 Sidecar 代理与其他服务通信：

1. **应用层**: Vigo 业务逻辑，专注于业务实现
2. **代理层**: Sidecar 处理所有网络通信
3. **控制层**: 服务网格控制平面管理配置

## 部署配置

### Docker Compose 部署

```yaml
version: '3.8'

services:
  app:
    image: your-org/vigo-app:latest
    container_name: vigo-app
    environment:
      - APP_ENV=production
    ports:
      - "8080:8080"
    networks:
      - mesh-network

  envoy:
    image: envoyproxy/envoy:v1.28-latest
    container_name: envoy-sidecar
    volumes:
      - ./envoy.yaml:/etc/envoy/envoy.yaml
    ports:
      - "9901:9901"  # Admin interface
    networks:
      - mesh-network
    depends_on:
      - app

networks:
  mesh-network:
    driver: bridge
```

### Envoy 配置示例

```yaml
# envoy.yaml
admin:
  address:
    socket_address:
      address: 0.0.0.0
      port_value: 9901

static_resources:
  listeners:
    - name: listener_0
      address:
        socket_address:
          address: 0.0.0.0
          port_value: 10000
      filter_chains:
        - filters:
            - name: envoy.filters.network.http_connection_manager
              typed_config:
                "@type": type.googleapis.com/envoy.extensions.filters.network.http_connection_manager.v3
                stat_prefix: ingress_http
                route_config:
                  name: local_route
                  virtual_hosts:
                    - name: local_service
                      domains: ["*"]
                      routes:
                        - match:
                            prefix: "/"
                          route:
                            cluster: vigo_app
                http_filters:
                  - name: envoy.filters.http.router
                    typed_config:
                      "@type": type.googleapis.com/envoy.extensions.filters.http.router.v3

  clusters:
    - name: vigo_app
      type: STRICT_DNS
      lb_policy: ROUND_ROBIN
      load_assignment:
        cluster_name: vigo_app
        endpoints:
          - lb_endpoints:
              - endpoint:
                  address:
                    socket_address:
                      address: app
                      port_value: 8080
```

## Istio 集成

### 安装 Istio

```bash
# 安装 Istio
istioctl install --set profile=demo

# 验证安装
kubectl get pods -n istio-system
```

### 注入 Sidecar

```bash
# 启用自动注入
kubectl label namespace default istio-injection=enabled

# 或手动注入
istioctl kube-inject -f deployment.yaml -o deployment-injected.yaml
```

### Vigo 应用部署

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vigo-app
  namespace: default
spec:
  replicas: 3
  selector:
    matchLabels:
      app: vigo-app
  template:
    metadata:
      labels:
        app: vigo-app
        version: v1
    spec:
      containers:
        - name: vigo-app
          image: your-org/vigo-app:latest
          ports:
            - containerPort: 8080
          env:
            - name: APP_ENV
              value: "production"
          readinessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /health
              port: 8080
            initialDelaySeconds: 15
            periodSeconds: 20
```

### 流量管理

#### VirtualService 配置

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: vigo-app-vs
spec:
  hosts:
    - vigo-app
  http:
    - match:
        - headers:
            version:
              exact: v2
      route:
        - destination:
            host: vigo-app
            subset: v2
    - route:
        - destination:
            host: vigo-app
            subset: v1
          weight: 90
        - destination:
            host: vigo-app
            subset: v2
          weight: 10
```

#### DestinationRule 配置

```yaml
apiVersion: networking.istio.io/v1beta1
kind: DestinationRule
metadata:
  name: vigo-app-dr
spec:
  host: vigo-app
  trafficPolicy:
    connectionPool:
      tcp:
        maxConnections: 100
      http:
        h2UpgradePolicy: UPGRADE
        http1MaxPendingRequests: 100
        http2MaxRequests: 1000
    loadBalancer:
      simple: ROUND_ROBIN
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
  subsets:
    - name: v1
      labels:
        version: v1
    - name: v2
      labels:
        version: v2
```

## 安全配置

### mTLS 加密

```yaml
# 启用全局 mTLS
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system
spec:
  mtls:
    mode: STRICT
```

### 授权策略

```yaml
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: vigo-app-auth
spec:
  selector:
    matchLabels:
      app: vigo-app
  rules:
    - from:
        - source:
            principals: ["cluster.local/ns/default/sa/api-gateway"]
      to:
        - operation:
            methods: ["GET", "POST"]
            paths: ["/api/*"]
```

  name: vigo-app-circuit-breaker
spec:
  host: vigo-app
  trafficPolicy:
    outlierDetection:
      consecutive5xxErrors: 5
      interval: 30s
      baseEjectionTime: 30s
      maxEjectionPercent: 50
      minHealthPercent: 30

### 重试配置

```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: vigo-app-retry
spec:
  hosts:
    - vigo-app
  http:
    - route:
        - destination:
            host: vigo-app
      retries:
        attempts: 3
        perTryTimeout: 2s
        retryOn: 5xx,reset,connect-failure,retriable-4xx
```

## 常见问题

### Q: Sidecar 代理影响性能吗？

A: 会有轻微影响（通常<10ms），但带来的可观测性、安全性、弹性能力远超这点开销。

### Q: 如何调试 Sidecar 问题？

A: 使用以下命令：

```bash
# 查看 Sidecar 配置
istioctl proxy-config all <pod-name>

# 查看 Envoy 日志
kubectl logs <pod-name> -c istio-proxy

# 检查连接
istioctl proxy-status
```

### Q: 本地开发如何测试？

A: 可以使用以下方案：
1. Docker Compose + Envoy
2. Minikube + Istio
3. Kind + Linkerd
4. 禁用 Sidecar 直接测试应用

### Q: 如何处理数据库连接？

A: 数据库连接通常绕过 Sidecar：

```yaml
# 排除数据库流量
traffic.sidecar.istio.io/excludeOutboundPorts: "3306,5432"
```

## 总结

Vigo 框架通过集成服务网格，提供了：

- ✅ **零代码侵入**: 无需修改业务代码即可获得服务网格能力
- ✅ **完整可观测性**: 链路追踪、指标监控、日志收集
- ✅ **强大流量管理**: 金丝雀发布、蓝绿部署、流量镜像
- ✅ **企业级安全**: mTLS 加密、身份认证、访问控制
- ✅ **弹性保障**: 熔断、降级、重试、超时控制

通过服务网格，Vigo 应用可以更加专注于业务逻辑，将服务间通信的复杂性交给基础设施层处理。
